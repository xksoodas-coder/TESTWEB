# خطة المتاجر متعددة المستأجرين (Multi-Tenant Storefronts) — على نمط Shopify

> الهدف: من **نشر واحد** (نفس الكود، نفس السيرفر، نفس قاعدة Turso) نقدّم **لكل زبون من زبائنك متجرًا خاصًا به**:
> - رابط خاص (Subdomain) مثل `boutique-ali.bigstore.dz`
> - أو مسار خاص مثل `bigstore.dz/s/ali`
> - أو **دومين خاص به** (Custom Domain) مثل `www.boutique-ali.com`
>
> وفي كل الحالات تظهر **منتجاته وتصنيفاته وإعداداته وحده فقط**، مربوطة بـ `store_id` الخاص به، **دون أي اختلاط** مع متاجر بقية الزبائن — تمامًا كما تفعل Shopify حيث كل التجار على بنية واحدة لكن كل متجر معزول.

---

## 0) الوضع الحالي (نقطة الانطلاق)

ما هو مبنيّ فعلًا ويُسهّل الخطة كثيرًا:
- **العزل بـ `store_id` موجود**: كل الاستعلامات (المنتجات، التصنيفات، الطلبيات، الإعدادات، تسجيل الدخول) مُقيّدة بـ `store_id` في توكن الجلسة.
- **توكن جلسة موقّع (HMAC)** يحمل `store_id`.
- **إعدادات لكل متجر** في جدول `bws_site_settings` (الألوان، الشريط، طريقة العرض، حجم الصفحة).
- **دخول الزبون** يتطلب حاليًا: رمز المتجر + اسم المستخدم + كلمة المرور.

**النقص الذي تعالجه هذه الخطة:**
1. تحديد المتجر **من الرابط نفسه** (Host أو المسار) بدل أن يكتب الزبون رمز المتجر يدويًا.
2. **سجل متاجر (Tenant Registry)** يربط: `slug` ↔ `custom_domain` ↔ `store_id`.
3. دعم **الدومينات الخاصة** بإصدار شهادات SSL تلقائيًا.
4. لوحة **مالك المنصة (أنت)** لإنشاء/تعطيل المتاجر وربط الدومينات.

---

## 1) المفاهيم الأساسية (كيف تفعلها Shopify ومن على شاكلتها)

Shopify (وكذلك Vercel/Notion/Substack…) تعتمد ثلاث طبقات لتحديد المستأجر:

| الطبقة | المثال | متى تُستعمل |
|---|---|---|
| **Subdomain للمنصّة** | `ali.myshopify.com` | يُعطى تلقائيًا لكل متجر فور إنشائه |
| **Path (مسار)** | `platform.com/s/ali` | بديل بسيط بلا إعداد DNS |
| **Custom Domain** | `www.boutique-ali.com` | عندما يملك الزبون دومينه ويربطه |

**القاعدة الذهبية:** *الرابط (Host/Path) هو الذي يحدّد المستأجر* → يُحوَّل إلى `store_id` → كل شيء بعد ذلك مُفلتر بـ `store_id`. لا مفاتيح مكتوبة يدويًا، ولا اختلاط ممكن.

**لتحدّي الدومينات الخاصة على نطاق واسع، النهج الأشهر والأقوى عالميًا:**
- **Cloudflare for SaaS — Custom Hostnames** (تستعمله Shopify فعليًا و آلاف منصّات SaaS). يتيح لزبونك توجيه دومينه إليك، ويصدر Cloudflare شهادة SSL لكل دومين تلقائيًا، وأنت تستقبل الطلب وتعرف الدومين من ترويسة `Host`.
- البديل المُدمج: **Vercel Domains API** (Wildcard + Custom Domains) — أبسط للبدء، كافٍ حتى عشرات/مئات المتاجر.

سنعتمد **مسارًا تدريجيًا**: Path → Subdomain (Wildcard على Vercel) → Custom Domains (Vercel أولًا، ثم Cloudflare for SaaS عند الكبر).

## 2) البنية المعمارية المقترحة

```
                         ┌─────────────────────────────────────────┐
   زبون التاجر  ─────▶   │  Edge / Middleware (تحديد المستأجر)        │
   (المتصفح)            │  يقرأ Host أو Path → يحلّ store_id        │
                         └───────────────┬─────────────────────────┘
                                         │ (store_id + theme)
                         ┌───────────────▼─────────────────────────┐
                         │  الواجهة (نفس ملفات bigwebstore)          │
                         │  تُحقن فيها هوية المتجر (store_id, theme)  │
                         └───────────────┬─────────────────────────┘
                                         │ Bearer token (store_id)
                         ┌───────────────▼─────────────────────────┐
                         │  /api/* (serverless) — كلها مفلترة        │
                         │  بـ store_id (موجود مسبقًا)               │
                         └───────────────┬─────────────────────────┘
                                         │
                         ┌───────────────▼─────────────────────────┐
                         │  Turso (قاعدة واحدة، عزل بـ store_id)     │
                         │  + جدول tenants (السجل الجديد)           │
                         └─────────────────────────────────────────┘
```

**مبدأ العزل (Tenant Isolation):** قاعدة واحدة + `store_id` على كل صف (نمط *Shared Database, Shared Schema* — نفس ما تستعمله Shopify/Slack في الأساس). أبسط وأرخص وأسهل صيانةً من قاعدة لكل زبون، وكافٍ تمامًا لحجمك.

---

## 3) جدول سجل المتاجر (Tenant Registry)

جدول جديد في Turso — **المصدر الوحيد للحقيقة** لربط الرابط بالمتجر:

```sql
CREATE TABLE IF NOT EXISTS bws_tenants (
    store_id       TEXT PRIMARY KEY,         -- نفس رمز المتجر المستعمل في كل الجداول
    slug           TEXT UNIQUE NOT NULL,     -- ali  → ali.bigstore.dz أو /s/ali
    custom_domain  TEXT UNIQUE,              -- www.boutique-ali.com (اختياري)
    domain_status  TEXT DEFAULT 'none',      -- none | pending | active (حالة تفعيل SSL/التحقق)
    display_name   TEXT,                     -- اسم المتجر للعرض
    is_active      INTEGER DEFAULT 1,        -- تعطيل المتجر (عدم الدفع/الإيقاف)
    plan           TEXT DEFAULT 'basic',     -- خطة الاشتراك (للمستقبل)
    created_at     TEXT,
    updated_at     TEXT
);
CREATE INDEX IF NOT EXISTS idx_tenants_slug   ON bws_tenants(slug);
CREATE INDEX IF NOT EXISTS idx_tenants_domain ON bws_tenants(custom_domain);
```

**ملاحظة مهمة:** `store_id` هو نفسه المستعمل في كل الجداول الحالية (المنتجات، التصنيفات…)، فلا حاجة لأي ترحيل للبيانات — فقط نضيف صفًّا في `bws_tenants` لكل زبون.

---

## 4) تحديد المستأجر (Tenant Resolution) — قلب النظام

دالة واحدة `resolveTenant(req)` تُرجع `store_id` بالترتيب التالي:

1. **Custom Domain:** إذا كانت ترويسة `Host` = أحد `custom_domain` في `bws_tenants` → ذلك المتجر.
2. **Subdomain:** إذا كان `Host` = `{slug}.bigstore.dz` → ابحث عن `slug`.
3. **Path:** إذا كان المسار يبدأ بـ `/s/{slug}` → ابحث عن `slug`.
4. وإلا → الصفحة العامة للمنصّة (تسويق/تسجيل تاجر جديد).

**التخزين المؤقت (Caching):** نتيجة الحل تُخزَّن في ذاكرة Edge/CDN لبضع دقائق (الخريطة slug/domain→store_id تتغيّر نادرًا) لتفادي استعلام Turso في كل طلب.

**على Vercel** يتم ذلك في `middleware.js` (Edge Middleware) الذي:
- يقرأ `Host`/المسار.
- يحلّ `store_id` (مع cache).
- يحقنه في ترويسة داخلية `x-store-id` (أو كوكي موقّع) تقرؤها الواجهة و `/api/*`.
- يعيد كتابة (rewrite) المسار `/s/{slug}/...` إلى الجذر مع الاحتفاظ بـ `store_id`.

---

## 5) أثر ذلك على تسجيل الدخول والجلسة

- **اليوم:** الزبون يكتب رمز المتجر يدويًا.
- **بعد التطبيق:** `store_id` يأتي **من الرابط** (آمن، يحدّده الـ middleware لا المستخدم). تسجيل الدخول يصبح **اسم المستخدم + كلمة المرور فقط**، و`/api/auth` يأخذ `store_id` من سياق المستأجر بدل جسم الطلب.
- التوكن الموقّع يبقى يحمل `store_id` (لا تغيير في منطق العزل في بقية الـ APIs).
- **حماية:** لا يستطيع زبون متجرٍ تمرير `store_id` متجرٍ آخر، لأن المصدر صار الـ middleware الموثوق وليس إدخال المستخدم.

---

## 6) الدومينات الخاصة (Custom Domains) — التفصيل

### المرحلة أ — عبر Vercel (للبدء، حتى مئات المتاجر)
1. الزبون يضيف عند مزوّد دومينه سجلًّا:
   - `CNAME www → cname.vercel-dns.com` (أو A record للجذر حسب تعليمات Vercel).
2. أنت (أو تلقائيًا عبر **Vercel Domains API**) تضيف الدومين إلى المشروع.
3. Vercel يتحقق ويُصدر شهادة **SSL تلقائيًا** (Let's Encrypt).
4. تُحدّث `bws_tenants.custom_domain` و`domain_status='active'`.
5. الـ middleware يطابق `Host` بالدومين → `store_id`.

### المرحلة ب — عبر Cloudflare for SaaS (للنطاق الواسع، نمط Shopify)
- استعمال **Custom Hostnames API**: لكل دومين زبون، تُنشئ Custom Hostname، وCloudflare:
  - يصدر شهادة SSL لكل دومين تلقائيًا (حتى آلاف الدومينات).
  - يوجّه الطلبات إلى أصلك (Origin) مع ترويسة `Host` الأصلية.
- مناسب جدًا عندما يصبح لديك عدد كبير من التجار بدومينات خاصة (هذا ما تفعله Shopify تحديدًا).
- التحقق من الملكية عبر `TXT`/`CNAME` (Cloudflare يوفّر تدفقًا جاهزًا).

> التوصية: ابدأ بـ Vercel (أسرع تشغيلًا)، وانتقل إلى Cloudflare for SaaS عند تجاوز حدود/تكلفة دومينات Vercel أو عند الحاجة لمرونة أكبر.

---

## 7) لوحة مالك المنصّة (Super-Admin) — خاصة بك أنت

لوحة منفصلة (أو قسم في برنامج الحاسوب) لإدارة المستأجرين:
- إنشاء متجر جديد: توليد `slug`، ربطه بـ `store_id` الموجود للزبون.
- ربط/فكّ **دومين خاص** + متابعة حالة التحقق وSSL.
- **تفعيل/تعطيل** متجر (مثلاً عند عدم الدفع → `is_active=0` فيظهر للزائر "هذا المتجر غير متاح حاليًا").
- إدارة الخطة/الاشتراك (`plan`) للمستقبل.
- لوحة إحصاءات سريعة لكل متجر.

**الأمان:** هذه اللوحة فوق صلاحية «مالك المنصّة» (مستوى أعلى من أدمن المتجر العادي `CanAccessWebsite`).

---

## 8) العزل والأمان (Isolation & Security)

- **كل استعلام يمرّ بـ `store_id`** المُستخرج من المستأجر (موجود مسبقًا) — لا اختلاط ممكن.
- **منع تسرّب البيانات:** الـ APIs لا تثق أبدًا بـ `store_id` قادم من جسم الطلب أو من المستخدم؛ تأخذه من التوكن/سياق المستأجر فقط.
- **حدود لكل مستأجر (Rate Limiting):** سقف للطلبات/الطلبيات لكل `store_id` (يوجد بالفعل سقف الطلبيات).
- **توقيع الجلسة** يبقى HMAC؛ يُضاف `tenantVersion` لإبطال الجلسات عند تعطيل متجر.
- **عناوين منفصلة:** كوكيز الجلسة تُربط بالدومين/المستأجر حتى لا تُستعمل جلسة متجر في متجر آخر.

---

## 9) الثيم والهوية لكل متجر

- موجود مسبقًا في `bws_site_settings` لكل `store_id` (ألوان، شريط، شعار، طريقة عرض).
- يُضاف للسجل: شعار، أيقونة المتصفح (favicon)، اسم المتجر في العنوان، ميتا SEO.
- الـ middleware/الواجهة تحقن الثيم باكرًا لتفادي وميض التحميل (FOUC).

---

## 10) SEO والأداء

- **sitemap.xml وrobots.txt لكل مستأجر** (مبنية ديناميكيًا حسب `store_id`).
- وسوم Open Graph لكل متجر (اسم/شعار/وصف).
- **Caching على الحافة (CDN)** للصفحات والـ APIs القابلة للتخزين، مع مفتاح يتضمّن `store_id`.
- صور المنتجات أصلًا على R2/CDN.

---

## 11) خطة التنفيذ على مراحل (Phased Rollout)

**المرحلة 1 — الأساس (Path-based):**
- جدول `bws_tenants` + دوال `resolveTenant`.
- دعم `‎/s/{slug}` على Vercel (rewrite) + حقن `store_id`.
- تعديل `/api/auth` ليأخذ `store_id` من المستأجر (مع إبقاء التوافق الخلفي مؤقتًا).
- لوحة super-admin بسيطة لإنشاء متجر/slug.

**المرحلة 2 — Subdomains:**
- ضبط **Wildcard Domain** `*.bigstore.dz` على Vercel.
- middleware يحلّ `{slug}.bigstore.dz`.
- توليد رابط تلقائي لكل زبون فور إنشائه.

**المرحلة 3 — Custom Domains (Vercel):**
- تدفّق ربط دومين + Vercel Domains API + متابعة SSL.
- تحديث `custom_domain`/`domain_status`.

**المرحلة 4 — التوسّع (Cloudflare for SaaS):**
- الانتقال للدومينات الخاصة عبر Custom Hostnames عند الكِبَر.

**المرحلة 5 — التسويق والفوترة:**
- صفحة منصّة عامة + تسجيل تاجر جديد ذاتي الخدمة.
- ربط الخطط/الاشتراك وتعطيل تلقائي عند عدم الدفع.

## 12) قائمة الملفات/التعديلات المتوقّعة (عند التطبيق لاحقًا)

> *(للتخطيط فقط — لن نلمسها الآن)*

- `bigwebstore/middleware.js` (جديد) — تحديد المستأجر على الحافة.
- `bigwebstore/api/_lib/tenant.js` (جديد) — `resolveTenant`, cache, جدول `bws_tenants`.
- `bigwebstore/api/auth.js` — أخذ `store_id` من المستأجر بدل جسم الطلب.
- `bigwebstore/api/_lib/session.js` — `tenantVersion`/ربط الكوكي بالمستأجر.
- `bigwebstore/api/tenants.js` (جديد) — إدارة المتاجر (super-admin).
- `bigwebstore/admin-*` — تبسيط الدخول (إخفاء حقل رمز المتجر عند معرفته من الرابط).
- `vercel.json` — rewrites للمسارات + إعداد wildcard.
- لوحة super-admin (صفحات جديدة) + تكامل اختياري مع برنامج الحاسوب.
- (لاحقًا) تكامل Cloudflare for SaaS Custom Hostnames API.

## 13) المخاطر والاعتبارات

- **DNS وSSL للدومينات الخاصة:** أكثر جزء يحتاج دعمًا تقنيًا للزبون؛ يُوصى بدليل خطوة-بخطوة + كشف حالة آلي.
- **التخزين المؤقت للمستأجر:** يجب إبطاله فور تغيير slug/domain/تعطيل المتجر.
- **حدود Turso:** قاعدة واحدة تكفي لآلاف المتاجر إن بقي العزل بـ `store_id` وفهارس مناسبة؛ المراقبة عند النمو.
- **الجلسات:** ضمان عدم تجاوز جلسة متجر إلى آخر (ربط الكوكي بالدومين).
- **التوافق الخلفي:** أثناء الانتقال، نُبقي دخول «رمز المتجر اليدوي» يعمل حتى تكتمل الروابط.

---

## 14) الخلاصة

البنية الحالية (عزل بـ `store_id` + إعدادات لكل متجر + توكن موقّع) **جاهزة 70% لفكرة Shopify**. المتبقّي:
1. **سجل متاجر** (`bws_tenants`).
2. **تحديد المستأجر من الرابط** (middleware: domain/subdomain/path).
3. **دومينات خاصة** (Vercel ثم Cloudflare for SaaS).
4. **لوحة مالك المنصّة** للإدارة والفوترة.

بهذا يحصل كل زبون من زبائنك على متجره الخاص — برابط أو دومين مستقل — معزولًا تمامًا بـ `store_id`، من نشر واحد وموقع واحد، تمامًا كما تعمل Shopify.
